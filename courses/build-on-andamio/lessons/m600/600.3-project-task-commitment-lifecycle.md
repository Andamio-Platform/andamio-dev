# The project task commitment lifecycle

## Before we start

M500.4 covered the course assignment lifecycle: four stable states, two outcomes. Projects follow the same shape but add a third outcome, escrow mechanics, and a heavier deposit. This lesson is the project-domain equivalent — the states between committing to a task and claiming a credential.

## The states

```
COMMITTED → ACCEPTED → credential claimed (deposit refund + reward)
           → REFUSED  → COMMITTED (retry with new evidence)
           → DENIED   → removed (deposit refund, reward clawed back)
```

| State | Set by | What it means |
|-------|--------|--------------|
| `COMMITTED` | Contributor's `project_join` TX reaches `updated` | Contributor has committed to a task. Evidence is on-chain. |
| `ACCEPTED` | Manager's `task_assess` TX reaches `updated` | Manager approved the work. Contributor can claim credential + reward. |
| `REFUSED` | Manager's `task_assess` TX reaches `updated` | Manager rejected the work. Contributor resubmits via a new commit TX. |
| `DENIED` | Manager's `task_assess` TX reaches `updated` | Permanent removal. Contributor is out. Reward returns to treasury. |

## Three outcomes

The difference from courses is the third outcome and what each outcome does to the money.

### Accept

The contributor's work meets the bar. On credential claim:
- The ~14.5 ADA state deposit returns to the contributor.
- The task reward transfers from escrow to the contributor, minus a 5% commission.
- A credential NFT is minted to the contributor's wallet.

Net: contributor gains ADA. Credential claims are the only transaction where the initiator ends richer.

### Refuse

"Try again." The contributor's evidence is cleared, and they can resubmit with a new `project_join` transaction. The task reward stays in escrow. The contributor's deposit stays locked. No limit on resubmissions.

Refuse is not punitive — it's iterative. A contributor who gets refused three times and then accepted earns the same credential and reward as someone accepted on the first try.

### Deny

Terminal. The contributor is permanently removed from the project. The task reward returns from escrow to the treasury. The contributor's ~14.5 ADA state deposit is refunded (deny removes the contributor, it doesn't confiscate their deposit). No credential is issued.

Deny exists because projects involve real money. A contributor acting in bad faith — plagiarizing, spamming, or submitting malicious code — needs to be removable, and the treasury needs to recover the reward for reassignment.

There is no un-deny. A denied contributor cannot rejoin the project.

## The deposit mechanics

The contributor's ~14.5 ADA deposit at `project_join` serves two purposes:

1. **Spam prevention.** Committing to a task costs real ADA. A contributor who commits to a task and never submits evidence has ADA locked in a pending state. The deposit creates a cost for occupying task slots.
2. **State UTxO funding.** The deposit pays for the on-chain UTxO that holds the contributor's state. When the credential is claimed, the UTxO is burned and the ADA returns.

The deposit is always refunded — on accept (via credential claim) and on deny. The contributor doesn't lose their deposit; they lose the opportunity cost of having it locked.

Compare with courses: the enrollment deposit is ~1.45 ADA, ten times smaller. The difference reflects the economic stakes. Courses are about learning (low stakes). Projects are about funded work (higher stakes, higher deposit).

## The escrow flow

Follow the ADA through the project lifecycle:

```
Treasury (owner funds it)
  → tasks_manage: reward moves to escrow
    → project_join: contributor deposits ~14.5 ADA
      → task_assess (accept): reward stays in escrow, waiting for claim
        → credential_claim: reward (minus 5%) → contributor, deposit → contributor
      → task_assess (refuse): reward stays in escrow, contributor retries
      → task_assess (deny): reward returns to treasury, deposit → contributor
```

At every step, the ADA is in exactly one place: treasury, escrow, or contributor's wallet. No ADA is created or destroyed — it moves between validators.

## Pending states and expiry

Same pattern as courses (M400.5). Between stable states, records pass through `PENDING_TX_*` states:

| Transition | Pending state | On TX expiry |
|-----------|--------------|--------------|
| First commit | `PENDING_TX_COMMIT` | **No revert** — no prior state |
| Manager assesses | `PENDING_TX_ASSESS` | Reverts to `COMMITTED` |
| Claim credential | `PENDING_TX_CLAIM` | Reverts to `ACCEPTED` |
| Contributor leaves | `PENDING_TX_LEAVE` | **No revert** — intentional departure |

The two non-reverting cases are the same as courses: `PENDING_TX_COMMIT` (nothing to revert to) and `PENDING_TX_LEAVE` (reverting would undo an intentional action).

## Batch assessment

Like course assessment, the `task_assess` transaction accepts a `task_decisions` array with mixed outcomes:

```json
{
  "task_decisions": [
    {"alias": "alice", "outcome": "accept"},
    {"alias": "bob", "outcome": "refuse"},
    {"alias": "carol", "outcome": "deny"}
  ]
}
```

One transaction, three contributors, three different outcomes. The manager reviews a batch and submits all decisions at once.

## Rubric

1. The reward stays in escrow — refuse doesn't move money. The contributor submits a new `project_join` transaction with revised evidence. No limit on retries.
2. The reward returns from escrow to the treasury. The contributor's ~14.5 ADA deposit is refunded. The contributor is permanently removed — they can't rejoin this project.
3. Projects involve real money (treasury, escrow, rewards). A bad-faith contributor occupying a funded task slot needs to be removable, and the treasury needs to recover the reward. Courses don't have this problem because there's no treasury and no financial incentive to game the system.
