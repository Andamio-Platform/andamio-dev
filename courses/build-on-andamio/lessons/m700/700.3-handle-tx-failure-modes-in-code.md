# Handle Andamio TX failure modes gracefully in code

## Before we start

M400.4 taught you the five failure modes and how to diagnose them. This lesson is about what to do about them in code. Each mode has a pattern: detect, surface, and either retry or escalate. The patterns are the same regardless of integration strategy — template, direct API, or CLI.

## The five modes, with code patterns

### 1. Build error

The Andamio API rejects the transaction body. HTTP 4xx with an error payload. Nothing has touched the chain.

**Pattern: throw immediately, surface the error, don't retry.**

```typescript
const response = await fetch(buildUrl, {
  method: "POST",
  headers: { "Content-Type": "application/json", ...authHeaders },
  body: JSON.stringify(txBody),
});

if (!response.ok) {
  const error = await response.text();
  throw new Error(`Build failed (${response.status}): ${error}`);
}
```

Build errors are always the caller's fault: bad alias, missing field, insufficient funds, wrong `course_id`. Retrying with the same body produces the same error. Fix the input and try again.

**Common causes and what to check:**

| Error | Likely cause | Fix |
|-------|-------------|-----|
| `alias field required` | Missing `alias` in body | Add alias |
| `insufficient funds` | Wallet doesn't have enough ADA | Fund wallet |
| `module not found` | Module not in PENDING_TX status | Run off-chain prep (M500.3) |
| `422 Unprocessable Entity` | Missing prerequisite credential | Earn the prerequisite first |

### 2. Sign error

Client-side. The signing key is missing, malformed, or the user declined in a browser wallet. No network call involved.

**Pattern: translate to a user-readable message, don't retry automatically.**

```typescript
function handleSignError(error: Error): string {
  const msg = error.message.toLowerCase();
  if (msg.includes("user declined") || msg.includes("user rejected")) {
    return "You declined to sign the transaction.";
  }
  if (msg.includes("no such file")) {
    return "Signing key file not found. Check your SKEY_PATH.";
  }
  if (msg.includes("insufficient") && msg.includes("fund")) {
    return "Insufficient funds in your wallet for this transaction.";
  }
  return `Signing failed: ${error.message}`;
}
```

Sign errors are always local. The user has to fix the wallet state or provide the right key before retrying.

### 3. Submit rejection

The signed transaction was broadcast to Cardano but the network rejected it. HTTP 400 from the submit endpoint.

**Pattern: surface the rejection reason, don't retry the same transaction.**

```typescript
const submitResponse = await fetch(submitUrl, {
  method: "POST",
  headers: { "Content-Type": "application/cbor" },
  body: signedTxBytes,
});

if (!submitResponse.ok) {
  const rejection = await submitResponse.text();
  throw new Error(`Submit rejected: ${rejection}`);
}
```

Submit rejections mean the chain state changed between build and submit — someone else spent a UTxO you were trying to use, or fees were miscalculated. The fix is to rebuild from scratch (new build → new sign → new submit). Never resubmit a rejected transaction.

### 4. Chain expiry

The transaction was submitted and registered, but never appeared on-chain within the timeout (2 hours). Status: `expired`.

**Pattern: poll with a timeout, treat `expired` as terminal, offer retry.**

```typescript
async function pollUntilTerminal(
  txHash: string,
  options: { interval: number; maxPolls: number }
): Promise<TxStatus> {
  for (let i = 0; i < options.maxPolls; i++) {
    if (i > 0) await sleep(options.interval);

    const status = await fetchTxStatus(txHash);

    if (status.state === "updated") return status;    // success
    if (status.state === "expired") return status;     // terminal — safe to retry
    if (status.state === "failed") return status;      // terminal — investigate
  }
  throw new Error(`Polling timeout after ${options.maxPolls} attempts`);
}
```

Reasonable defaults: 6-second interval, 150 polls (~15 minutes). On preprod, most transactions reach `updated` within 90 seconds. An `expired` status means the transaction never landed — the entire operation is safe to retry from build.

### 5. Off-chain sync failure

The transaction landed on-chain (`confirmed_at` is set) but the Andamio API couldn't update its database after 5 retries. Status: `failed` with a `failure_reason`.

**Pattern: detect the confirmed-but-failed state, treat as operational success for UX.**

```typescript
if (status.state === "failed" && status.confirmed_at !== null) {
  // On-chain succeeded, off-chain sync failed
  // The credential/enrollment/assessment IS on-chain
  // The API will heal itself — show success with a note
  return {
    success: true,
    note: `Transaction confirmed on-chain (${status.tx_hash}). ` +
          `Database sync pending — data will appear shortly.`,
  };
}

if (status.state === "failed" && status.confirmed_at === null) {
  // Never reached chain — true failure
  return { success: false, error: status.last_error };
}
```

The `failure_reason` field tells you what the sync couldn't find: `MODULE_NOT_FOUND`, `TASK_NOT_FOUND`, etc. The fix is operational — create the missing off-chain record and the API retries automatically.

This is the subtlest failure mode. The on-chain write succeeded. The credential exists. Only the API's mirror is stale. Don't tell the user the operation failed.

## The defensive pattern

After any terminal status (`expired`, `failed`), fetch the record's current state before deciding what to do:

```typescript
async function handleTerminalStatus(status: TxStatus): Promise<Action> {
  if (status.state === "expired") {
    return { action: "retry", reason: "Transaction never landed on-chain." };
  }

  if (status.state === "failed" && status.confirmed_at) {
    return { action: "wait", reason: "On-chain succeeded. DB sync pending." };
  }

  if (status.state === "failed") {
    return { action: "investigate", reason: status.failure_reason ?? status.last_error };
  }

  return { action: "success" };
}
```

Don't assume. Read the status, check `confirmed_at`, and route accordingly.

## CLI equivalents

If you're using the CLI-first strategy, the same patterns apply at the shell level:

```bash
# Run with timeout, capture exit code
if ! andamio tx run "$ENDPOINT" --body "$BODY" --skey "$SKEY" --tx-type "$TYPE" --timeout 300; then
  # Check what happened
  STATUS=$(andamio tx status "$TX_HASH" --output json 2>/dev/null)
  STATE=$(echo "$STATUS" | python3 -c "import json,sys; print(json.load(sys.stdin).get('state','unknown'))")

  case "$STATE" in
    expired)  echo "Expired — safe to retry" ;;
    failed)   echo "Failed — check failure_reason" ;;
    *)        echo "Unexpected state: $STATE" ;;
  esac
fi
```

The CLI's exit codes: 0 = success (`updated`), 1 = failure (any terminal error). Check `tx status` for details.

## Your turn

A user runs a `credential_claim` transaction. Polling shows `state: failed`, `confirmed_at: "2026-04-05T12:00:30Z"`, `failure_reason: "MODULE_NOT_FOUND"`.

1. Did the credential get minted on-chain?
2. What should the UI show the user?
3. What's the operational fix?

Write your answers before checking the rubric.

## Rubric

1. **Yes** — `confirmed_at` is set, which means the on-chain transaction succeeded. The credential NFT is in the user's wallet.
2. **Success with a note** — "Your credential was minted successfully. Some data may take a moment to appear." Don't show an error. The user's credential exists.
3. **Create or fix the module record** in the Andamio API so the sync can complete. The `MODULE_NOT_FOUND` reason means the API couldn't find the off-chain module to update. Once the record exists, the API retries the sync automatically.

## What you just did

You know the code pattern for each of the five failure modes: throw on build errors, translate sign errors, rebuild on submit rejections, retry on expiry, and treat confirmed-but-failed as success. The patterns work across all three integration strategies.
