# API keys and User JWTs

## Before we start

Your alias exists on-chain. Now it needs to be authenticated to the Andamio API before the API will act on your behalf. There are two methods, designed for different use cases.

## The two methods

|  | `X-API-Key` | User JWT |
|--|-------------|----------|
| **What it is** | Long-lived opaque string from the developer portal | Short-lived signed token from wallet auth |
| **Sent as** | `X-API-Key: ak_xxxx` header | `Authorization: Bearer eyJ...` header |
| **Grants** | Read access only | Read + write for the alias it was issued to |
| **Lifetime** | Permanent until revoked | ~24 hours, then re-sign |
| **Represents** | Your developer account ("who's asking") | A specific alias ("who's acting") |

On a write request, you send **both** headers:

```http
POST /v2/tx/course/teacher/modules/manage
Authorization: Bearer eyJhbGciOi...(a long JWT)...
X-API-Key: ak_xxxxxxxxxxxxxxxxxxxxxxxx
```

Don't send a JWT alone — the API's rate limiting runs off the API key, so JWT-only requests get rejected.

## When to use each

| Scenario | Headers |
|----------|---------|
| Read-only (list courses, query credentials, chart a treasury balance) | API key only |
| Write on behalf of an alias (submit evidence, create a course, assess a task) | API key + User JWT |
| Multi-tenant app where users act independently | API key (yours, the operator) + User JWT (one per user) |

Mental model: **the API key says "who's asking" and the JWT says "who's acting."** For a solo developer, they're the same person. For a multi-tenant deployment, they're different entities.

## Why two instead of one

- **Performance.** API keys are cheap to verify. Running a cryptographic signature check on every read would slow the API for no benefit.
- **Multi-tenancy.** The developer account (operator) and the end user (actor) are not always the same entity. Separating them lets the API apply rate limits to you and attribute on-chain actions to your users without conflating the two.

## Your turn

Which auth headers does each request need?

1. A script that fetches all public courses and writes titles to a CSV.
2. A CLI command that submits evidence for an assignment.
3. A dashboard that reads a project's treasury balance every 30 seconds.
4. A mobile app that lets its users mint their own access tokens from the UI.

Write your answers before checking the rubric.

## Rubric

1. **API key only.** Pure read — no alias acting.
2. **API key + User JWT.** Write (submitting evidence) — JWT identifies the submitter.
3. **API key only.** Read, even though the data is project-specific.
4. **API key + User JWT, one per user.** Multi-tenant case: the app sends its API key on every request, each user gets their own JWT when they sign in with their wallet.

Scenario 4 is the subtlest — re-read "Why two" if it tripped you up.

## What you just did

You can distinguish the two auth methods by what they represent: API keys identify developers, User JWTs identify aliases. You know when each is needed alone and when they're needed together.
