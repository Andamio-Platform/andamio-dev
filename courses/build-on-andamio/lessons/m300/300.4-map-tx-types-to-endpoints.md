# Map each tx_type to its build endpoint

## Before we start

Last lesson in Module 300. You've seen the read side from three angles. This lesson is the write catalog — the complete menu of things you can do on the protocol. You won't write anything yet; Module 400 uses this map to run transactions.

## What a tx_type is

Every write operation has a string identifier called its `tx_type`. You saw `access_token_mint` in M200.1. The `tx_type` tells the Andamio API which kind of transaction to build and how to track it through the state machine. It's also how you look up the right build endpoint.

If something isn't represented by a `tx_type`, you can't do it as a transaction.

## Getting the list

```bash
andamio tx types --output json
```

Each entry in the response:

| Field | What it tells you |
|-------|-------------------|
| `tx_type` | String identifier (`course_create`, `assignment_submit`, etc.) |
| `endpoint` | API path to POST when building this transaction |
| `category` | Domain: `course`, `project`, or `general` |
| `description` | What the type does |
| `updates_db` | Whether the API writes an off-chain row after on-chain confirmation |

## The updates_db split

The `updates_db` field splits the catalog into two groups:

**`updates_db: true`** — transaction creates or modifies off-chain records alongside its on-chain effect. `updated` status means "on-chain confirmed AND off-chain synced."

| tx_type | What it does |
|---------|-------------|
| `course_create` | Create a new course |
| `modules_manage` | Add, update, or remove modules |
| `teachers_update` | Update course teacher list |
| `assignment_submit` | Commit student assignment evidence |
| `assessment_assess` | Teacher accepts or refuses an assignment |
| `project_create` | Create a new project |
| `project_join` | Contributor commits to a task |
| `managers_manage` | Update project manager list |
| `tasks_manage` | Add or remove project tasks |
| `task_submit` | Submit or update task evidence |
| `task_assess` | Manager accepts, refuses, or denies a submission |
| `project_credential_claim` | Contributor claims their reward |

**`updates_db: false`** — on-chain only, no off-chain row written. `updated` means "on-chain confirmed, nothing more to sync."

| tx_type | What it does |
|---------|-------------|
| `access_token_mint` | Mint an access token (M200.1) |
| `course_enroll` | Enroll without an off-chain commitment |
| `credential_claim` | Claim a course credential NFT |
| `blacklist_update` | Update project contributor blacklist |
| `treasury_fund` | Add ADA to a project's on-chain treasury |

In practice, you always wait for `updated` either way. The distinction matters when debugging: knowing whether the API is waiting on an off-chain sync tells you which kind of failure you might see.

## The write workflow

When you're about to run a write in M400 or later:

1. Identify what you want to do → "create a course"
2. Look up the `tx_type` → `course_create`
3. Get the build endpoint from the same response → `/v2/tx/instance/owner/course/create`
4. Build the request body (the spec tells you the shape)
5. `andamio tx run <endpoint> --body-file ... --skey ... --tx-type <tx_type>`

`andamio tx types` is steps 2 and 3. Run it whenever you're about to write a transaction you haven't run before.
