# Why `updated` is the only safe TX status for refetching data

## Before we start

**`updated` is the only status that means your transaction is done and the data it affected is safe to read.** Every other status means "still working" or "something went wrong." Code that refetches on `confirmed` will work most of the time and silently break some of the time — the worst kind of bug.

## The five statuses

| Status | Meaning | Action |
|--------|---------|--------|
| `pending` | Registered with the API, waiting for on-chain confirmation | Keep waiting |
| `confirmed` | Cardano confirmed on-chain, but API hasn't finished syncing off-chain data | Keep waiting |
| `updated` | On-chain confirmed AND off-chain sync complete | **Safe to refetch** |
| `failed` | On-chain confirmed, but off-chain sync failed permanently | Handle error (M400.4) |
| `expired` | Two hours passed, TX never landed on-chain | Safe to retry |

## The trap

`confirmed` looks like "done." It isn't. It means "Cardano has the transaction, but the Andamio API — the thing your app queries — hasn't reflected the transaction's effects yet."

Concrete bug:

```javascript
// Submit a teacher's acceptance of a student assignment
const result = await runTransaction("assessment_assess", body);

// BUG: waiting for "confirmed" instead of "updated"
await waitForStatus(result.tx_hash, "confirmed");

// Refetch shows pre-acceptance state on busy networks
const credentials = await fetchStudentCredentials(studentAlias);
```

This works most of the time. When the off-chain sync is fast (usually), it finishes before the refetch. Then during a demo on a busy network, the refetch runs while sync is still in progress, and the student's profile shows "not yet accepted" even though the on-chain TX confirmed.

Intermittent. Impossible to reproduce locally. Depends on network timing. The fix is one word:

```javascript
await waitForStatus(result.tx_hash, "updated");
```

Now the refetch only runs after the API has synced. Guaranteed correct under any load.

## Why the API has a separate sync step

- **Performance.** Querying Cardano on every request would be slow. The API caches a processed chain view in its own storage for fast reads.
- **Off-chain data.** Titles, content, draft states (M300.1) don't live on-chain. When a TX confirms, some off-chain fields need updating (module status `PENDING_TX → ON_CHAIN`, commitment evidence hashes, teacher list rebuilds). That work takes real time.

The `confirmed → updated` gap is exactly that work. Usually 1–2 seconds on preprod. Under heavy load, up to 10+ seconds. Never long — just long enough to break code that doesn't wait.

## The rule

**Always wait for `updated` before reading data your transaction affected.** The only exception: if you're reading data the transaction *didn't* affect (a different student's credentials, a different course, an unrelated treasury balance), you can read at any time.

## Rubric

**Bug scenario.** A strong answer: developer awaits `confirmed` then refetches → works most of the time because sync is fast → fails on busy networks when refetch runs before sync → returns pre-transaction state → fix is `"updated"` instead. Key insight: `confirmed` = on-chain-done but not API-done.

**Exception.** Reading data unaffected by your transaction. Examples: you accepted student A's assignment and want to read student B's unrelated credentials; you created a course and want to list other courses; you submitted evidence and want to read a project treasury balance. In all cases, the data isn't being synced, so timing doesn't matter.
